# Обзор системы доступов

Модуль **Access** управляет правами доступа менеджеров ко всем админ-разделам сайта. Вместо жёстко зашитого списка "кто имеет доступ куда" система использует **роли** и **индивидуальные права**.

## Основные понятия

**Разрешение (Permission)** — одна сущность в системе, например `orders.orders` или `competitors.dracar.products`. У каждого разрешения есть набор действий: `view` (просмотр), `edit` (редактирование), `delete` (удаление), `import` (импорт) и т.д.

**Роль (Role)** — именованный набор разрешений. Например, роль `seller` может иметь `view` и `edit` на заказы, но не иметь доступа к складу.

**Пользователь (User)** — менеджер сайта. У каждого пользователя может быть одна роль (или ни одной).

**Override (Индивидуальное право)** — точечное исключение из правил роли. Например, менеджер с ролью `seller` может дополнительно получить доступ к одному конкретному разделу, которого нет в роли.

## Как всё работает вместе

1. Админ создаёт **роли** на странице *Roles*
2. На странице *Matrix* админ расставляет разрешения роли — какие действия разрешены
3. На странице *Users* админ назначает роли менеджерам
4. Для исключительных случаев — добавляет **override** конкретному менеджеру

Когда менеджер пытается открыть раздел, система проверяет:

1. Он суперадмин? → Да → доступ
2. Есть override на запрет (revoke)? → Да → отказ
3. Есть override на разрешение (grant)? → Да → доступ
4. Роль разрешает? → Да → доступ
5. Иначе → отказ

## Когда использовать override

**Роль** — это **базовый** набор прав для группы менеджеров. Если одному менеджеру нужно дать временный доступ к чему-то — не создавайте отдельную роль, а добавьте ему override.

Пример: есть роль `seller` для 15 менеджеров. Нужно, чтобы именно Иван Петров мог смотреть отчёт по финансам, которого нет в `seller`. Вместо создания новой роли — открываете его в Users, кликаете бейдж `view` на `finances.reports` — добавляется синий override-grant. Все остальные 14 менеджеров остаются с неизменными правами.
